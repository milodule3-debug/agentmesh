❌ Problem: Agents Repeat the Same Mistakes

Without Learning

🔍 Researcher tries to access a site → fails

🔍 Tries the same thing again → fails again

🔍 Same thing a third time → crash

Same error pattern, zero progress

With Recursive Learning

🔍 Researcher tries → fails

🧠 Analyzes why → learns a lesson

🔍 Tries differently → success!

Every failure = a new lesson

Key idea: Instead of failing the same way repeatedly, the agent analyzes its errors and changes its behavior for next time.

📊 Two Levels of Learning

🧑‍🎓 Level 1: Individual Agent

  • Agent analyzes ITS OWN failures
  • Extracts concrete, specific rules
  • Modifies ITS OWN system prompt
  • Example: "Always check if file exists before reading"

Like a student learning from their test results

👔 Level 2: Orchestrator (Manager)

  • Analyzes failures across the WHOLE team
  • Changes task assignment STRATEGY
  • Optimizes budgets (time, tokens)
  • Example: "Don't give web search tasks to the agent that keeps failing"

Like a manager optimizing their team

These two levels work simultaneously — agents learn individually, and the manager learns how to assign work better.

🧠 Agent Memory — 4 Layers

Think of it like the human brain — each layer stores a different type of information:

📖

Episodic Memory

RAW notes about every task — what it did, how many tokens it burned, what crashed. NEVER summarized, kept verbatim.

💡

Semantic Memory

Learned rules and lessons — "Always check the API response before parsing." This gets injected into every future task.

📊

Procedural Memory

Statistics on which tools work well and which don't. E.g.: "web_search = 90% success, http_get = 60% success."

📝

Working Memory

Current scratchpad — what the agent is doing RIGHT NOW. Saved to disk, survives crashes.

🚦 Acceptance Gate — Lesson Filter

Not every lesson passes! The system has strict filters:

❌ Rejected Lessons

  • "Be more careful" — too generic
  • "Check everything" — doesn't say WHAT
  • "Try again" — no rule given
  • Shorter than 30 characters

✅ Accepted Lessons

  • "Check if 'choices' key exists in JSON before accessing"
  • "Log the full API response on any error"
  • "Use try/except for all HTTP calls"
  • Concrete, measurable, testable

Why? Research (Tsinghua 2026) shows that narrowing the attempt loop helps, but broadening hurts. Better 1 good lesson than 10 bad ones.
